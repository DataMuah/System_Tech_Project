# 멀티채팅 서버 프로젝트 최종 보고서

**Rust + Tokio 기반 단일 채팅방 — 500명 동시접속 구현 및 검증**

---

## 1. 프로젝트 개요

### 1.1 무엇을 만들었나

500명이 동시에 입장한 단톡방을 가정하고, 모두가 각자 다른 메시지를 보냈을 때 서버가 그 트래픽을 무너지지 않고 받아내는지 확인하는 것이 이번 프로젝트의 목표였다. 이를 위해 Rust 언어와 Tokio 비동기 런타임을 사용해 다음 세 가지 바이너리를 만들었다.

- **server** — TCP 8080 포트에서 클라이언트를 받아 메시지를 브로드캐스트하는 본체.
- **stress_test** — 500명을 동시에 붙여 보고 결과를 수치로 보여주는 부하 테스터.
- **monitor** — 서버 코드를 건드리지 않고 트래픽을 실시간 시각화하는 TUI 도구.

### 1.2 왜 Rust + Tokio 인가

전통적인 C/Java 스타일로 클라이언트 1명당 OS 스레드 1개를 띄우면, 500명이면 1 GB에 가까운 스택 메모리를 잡아먹고 컨텍스트 스위칭으로 지연이 누적된다. Tokio의 비동기 태스크는 수 KB 수준의 가벼운 상태 머신이라 같은 일을 수십 MB 안에서 끝낼 수 있고, Rust의 소유권 모델 덕분에 데이터 경합도 컴파일러가 잡아 준다. 동시성 처리량 + 메모리 안전성을 한 번에 잡기에 적합한 조합이라고 판단했다.

### 1.3 핵심 요구사항

다음을 만족하는 서버를 만드는 것이 출발점이었다.

```
✓ 단일 채팅방 실시간 메시지 브로드캐스트
✓ 닉네임 기반 클라이언트 식별
✓ 입장 / 퇴장 알림
✓ HH:MM:SS 타임스탬프 포함 메시지 포매팅
✓ 자기 메시지 echo 차단
✓ 500명 동시 접속에서 연결 실패 0건
```

## 2. 시스템 설계

### 2.1 전체 구조

서버는 단 하나의 `TcpListener`가 들어오는 모든 연결을 받고, 각 연결마다 비동기 태스크 하나를 띄워 처리한다. 모든 태스크는 단 하나의 `broadcast::channel`을 공유하고, 이 채널이 메시지를 모든 클라이언트에게 한 번에 뿌려 준다.

```
                     ┌──────────────────────────────────────────────────────────┐
                     │                      CHAT SERVER                         │
  Client A ──TCP──▶  │  ┌────────────────┐                                      │
  Client B ──TCP──▶  │  │  TcpListener   │  accept() loop                       │
  Client C ──TCP──▶  │  │  0.0.0.0:8080  │                                      │
      ...            │  └───────┬────────┘                                      │
                     │          │ tokio::spawn() per connection                 │
                     │          ▼                                               │
                     │  ┌─────────────────────┐   Arc<RwLock<HashMap>>          │
                     │  │   handle_client()   │ ◀▶  { addr → nickname }         │
                     │  │  into_split()       │                                 │
                     │  │  ┌────────┐         │   Arc<AtomicUsize>              │
                     │  │  │ reader │──read──▶│ ◀▶  total_connected             │
                     │  │  └────────┘         │           ↓                     │
                     │  │  ┌────────┐         │  ┌──────────────────────┐       │
                     │  │  │ writer │◀─write──│  │ broadcast::channel   │       │
                     │  │  └────────┘         │  │     (cap = 10,000)   │       │
                     │  └─────────────────────┘  └──────────────────────┘       │
                     │                                       ↑                  │
                     │                    fan-out to all subscribers            │
                     │          ┌────────────────────────────┘                  │
                     │       rx.recv()  rx.recv()   rx.recv()                   │
                     │       Client B   Client C   Client D  ...                │
                     └──────────────────────────────────────────────────────────┘
```

### 2.2 스레드 대신 비동기 태스크를 쓴 이유

500명 처리 시 자원 사용량을 단순히 비교해 보면 차이가 분명하게 드러난다.

```
  전통적 Thread-per-Connection           Tokio Async Task (본 프로젝트)
  ─────────────────────────────          ───────────────────────────
  클라이언트 500명                        클라이언트 500명
       │                                       │
       ├─ Thread 1   (2MB 스택)                ├─ async task 1   (~수 KB)
       ├─ Thread 2   (2MB 스택)                ├─ async task 2
       ...                                     ...
       └─ Thread 500 (2MB 스택)                └─ async task 500
                                                       │
  총 메모리 : ~1 GB                       Tokio 워커 풀 (CPU 코어 수)
  컨텍스트 스위칭 : 잦음                    총 메모리 : ~수 MB
                                          컨텍스트 스위칭 : 최소
```

태스크는 OS 스레드와 분리된 협력적 다중화 단위이기 때문에, 500개를 띄워도 워커 스레드는 CPU 코어 수만큼만 돌아간다. 이 차이가 본 프로젝트가 무리 없이 500명을 받아낸 핵심 이유다.

### 2.3 메시지가 흐르는 길

한 클라이언트가 입력한 줄은 단 한 번만 `Message`로 포장되어 broadcast 채널로 들어가고, 채널 내부의 링 버퍼를 `Arc`로 공유하기 때문에 수신자가 500명이든 1,000명이든 본문 자체는 추가로 복제되지 않는다.

```
클라이언트 A 가 "안녕" 입력
        │
        ▼
   [TCP Socket]
        │  BufReader::read_line()
        ▼
  handle_client() 수신 루프
        │  tx.send(Message { from: "A", body: "안녕", ... })
        ▼
  broadcast::channel  ←── 단 1회 복사, Arc 로 모든 Receiver 공유
        │
        ├──▶  Receiver B → write_task → TCP Socket → Client B   ✅
        ├──▶  Receiver C → write_task → TCP Socket → Client C   ✅
        ├──▶  Receiver A → write_task →  msg.addr == A → skip   🚫 echo 없음
        └──▶  ...
```

### 2.4 공유 상태를 어떻게 관리했나

서버에서 공유되는 상태는 두 가지뿐이고, 각각에 가장 잘 맞는 도구를 골랐다.

**접속 중인 클라이언트 목록**은 `Arc<RwLock<HashMap<SocketAddr, String>>>`로 관리한다. 입·퇴장 시점에만 쓰기가 발생하고 평소엔 읽지도 않으니, 다중 읽기를 허용하는 `RwLock`이 `Mutex`보다 적합했다.

**누적 카운터**(연결 수, 메시지 수)는 `Arc<AtomicUsize>`로 단순하게 처리했다. 락 없이 `fetch_add(Relaxed)` 한 줄이면 끝이고, 수백 개 태스크가 동시에 갱신해도 경합이 거의 없다.

## 3. 구현 상세

### 3.1 메시지 자료구조

```rust
type ClientMap = Arc<RwLock<HashMap<SocketAddr, String>>>;

#[derive(Clone, Debug)]
struct Message {
    from:      String,      // 발신자 닉네임
    addr:      SocketAddr,  // echo 차단 판별용
    body:      String,      // 메시지 본문
    timestamp: String,      // HH:MM:SS
}
```

`addr`은 화면에 보이지 않고, 오직 "이 메시지를 자기 자신에게는 다시 보내지 말자"는 echo 차단용으로만 쓴다. broadcast 채널을 통과해야 하므로 `Clone`은 필수다.

### 3.2 진입점 `main()`

```rust
#[tokio::main]
async fn main() {
    let listener = TcpListener::bind("0.0.0.0:8080").await.expect("포트 바인딩 실패");

    // broadcast: 모든 클라이언트가 subscribe()로 Receiver 획득 (용량 10,000)
    let (tx, _rx) = broadcast::channel::<Message>(10_000);
    let tx = Arc::new(tx);

    let clients: ClientMap   = Arc::new(RwLock::new(HashMap::new()));
    let total_connected      = Arc::new(AtomicUsize::new(0));
    let total_messages       = Arc::new(AtomicUsize::new(0));

    loop {
        // 실제 코드는 match 로 accept 오류를 처리하고 다음 연결을 기다린다
        let (socket, addr) = match listener.accept().await {
            Ok(v)  => v,
            Err(_) => continue,
        };
        let tx = tx.clone();
        let rx = tx.subscribe();
        // OS 스레드를 새로 만들지 않고 경량 async task 로 분기
        tokio::spawn(async move {
            handle_client(socket, addr, tx, rx, clients.clone(), ...).await;
        });
    }
}
```

accept 루프에서 가장 중요한 부분은 마지막 `tokio::spawn`이다. 새 OS 스레드를 만드는 게 아니라 비동기 태스크 하나를 띄우는 것뿐이라 비용이 거의 없다.

### 3.3 클라이언트 핸들러: deadlock 을 구조적으로 막기

소켓 하나를 그대로 쓰면, 같은 태스크 안에서 "읽기를 기다리면서 동시에 쓰기를 처리"해야 하는 상황이 만들어지고 결국 자기 자신을 기다리며 멈추는 self-deadlock이 발생한다. 이 문제를 코드 차원이 아니라 **타입 차원**에서 차단하기 위해 `into_split()`으로 소켓을 둘로 쪼갰다.

```rust
let (reader, mut writer) = socket.into_split();
//        │                    └── 쓰기 전용 → write_task 로 소유권 이동
//        └── 읽기 전용 → 현재 태스크에서 사용
```

쪼갠 다음에는 둘이 서로를 모르고 일하면 된다.

**송신 태스크** — broadcast로 들어온 메시지를 자기 클라이언트 소켓에 쓴다.

```rust
let write_task = tokio::spawn(async move {
    loop {
        match rx.recv().await {
            Ok(msg) => {
                if msg.addr == addr { continue; }   // 자기 메시지 echo 차단
                let line = format!("[{}] {}: {}\n",
                                   msg.timestamp, msg.from, msg.body);
                if writer.write_all(line.as_bytes()).await.is_err() { break; }
            }
            Err(RecvError::Lagged(n)) => eprintln!("⚠️  {addr} lagged {n}"),
            Err(_) => break,
        }
    }
});
```

**수신 루프** — 클라이언트 소켓에서 읽어 broadcast로 흘려보낸다.

```rust
loop {
    line.clear();
    match buf_reader.read_line(&mut line).await {
        Ok(0) => break,                                                  // EOF
        Ok(_) => {
            let body = line.trim().to_string();
            if body.is_empty() { continue; }
            msg_count.fetch_add(1, Ordering::Relaxed);
            let _ = tx.send(Message {
                from: nick.clone(), addr, body, timestamp: now(),
            });
        }
        Err(e) => { eprintln!("읽기 오류: {e}"); break; }
    }
}
```

**퇴장 처리** — 송신 태스크를 abort 하고, 목록에서 빼고, 다른 사용자에게 퇴장을 알린다.

```rust
write_task.abort();
clients.write().await.remove(&addr);
let count = connected.fetch_sub(1, Ordering::Relaxed) - 1;
let _ = tx.send(Message {
    from: "SERVER".into(),
    addr,
    body: format!("[{nick}] 퇴장 (현재 {count}명)"),
    timestamp: now(),
});
```

### 3.4 broadcast 채널 용량은 왜 10,000?

너무 작으면 폭주 트래픽에서 메시지가 밀려 손실되고, 너무 크면 메모리만 잡는다. 클라이언트 500명이 한꺼번에 메시지를 보내는 burst 상황을 가정하고, 송신 태스크가 잠깐 지연돼도 수백 ms 분량은 안전하게 받쳐 줄 수 있도록 10,000으로 잡았다. 만약 이 한도를 넘으면 `RecvError::Lagged(n)`이 발생하므로, 그때는 콘솔에 경고를 찍고 추후 튜닝의 단서로 쓴다.

## 4. 부하 테스트

### 4.1 stress_test 의 설계 의도

500명을 한꺼번에 `connect`로 던지면 운영체제의 accept 큐가 즉시 포화되어 일부 연결은 실패한다. 이걸 피하려고 두 가지 장치를 넣었다.

**(1) 50개씩 10 배치, 50 ms 간격으로 발사** — 큐 포화 회피.
**(2) 일시적 실패는 백오프 후 최대 5회 재시도** — 자잘한 흔들림 흡수.

```
                    stress_test 실행
                          │
         ┌────────────────┼─────────────────┐
    Batch 1 (50)     Batch 2 (50)  ...  Batch 10 (50)
         │            [50 ms 대기]            │
         └─────── 한 번에 500개 연결하지 않음 ───┘
                    → accept() 큐 과부하 방지

  각 task 내부:
  ┌──────────────────────────────────────────────────┐
  │  connect_with_retry(addr, 최대 5회)              │
  │  닉네임 전송  "client_NNNN\n"                    │
  │  ┌────────────────┐   ┌──────────────────────┐   │
  │  │  recv_task     │   │  송신 루프            │   │
  │  │  RECV++        │   │  N_MESSAGES 회 송신   │   │
  │  └────────────────┘   │  10 ms 간격           │   │
  │                       │  SENT++              │   │
  │                       └──────────────────────┘   │
  │   3초 대기 후 recv_task.abort()                  │
  └──────────────────────────────────────────────────┘
```

### 4.2 결과를 어떻게 집계하나

전역 카운터 4개를 `AtomicUsize`로 잡고 각 태스크가 자유롭게 갱신한다. 락이 없으므로 카운터 자체가 병목이 되지 않는다.

```rust
static SENT:      AtomicUsize = AtomicUsize::new(0);  // 총 송신
static RECV:      AtomicUsize = AtomicUsize::new(0);  // 총 수신
static CONNECTED: AtomicUsize = AtomicUsize::new(0);  // 연결 성공
static FAILED:    AtomicUsize = AtomicUsize::new(0);  // 연결 실패

SENT.fetch_add(1, Ordering::Relaxed);
```

이론상의 총 수신 건수는 다음 식으로 미리 계산할 수 있다.

```
이론상 총 수신 = (총 송신) × (전체 클라이언트 − 1)
               = (500 × 5) × 499
               = 1,247,500 건
```

단, 본 테스터는 메시지 송신을 마친 뒤 3초만 더 기다리고 끊기 때문에, 늦게 합류한 클라이언트가 보낸 메시지는 일부만 잡힌다. 이 부분은 5장에서 다시 다룬다.

## 5. 운영 시각화: TUI 모니터

수치만 보면 서버가 정말 정상인지 직관적으로 와닿지 않는다. "500명이 떠드는 단톡방을 눈으로 본다"는 시각화가 필요해서 monitor 바이너리를 별도로 만들었다.

핵심 아이디어는 **"서버 코드는 한 줄도 건드리지 않는다"** 는 것이다. 모니터는 그냥 또 한 명의 클라이언트로 8080에 접속한다.

```
설계 원칙: 모니터 = 또 한 명의 클라이언트

  ┌──────────────────────┐
  │      server (8080)   │  ← main.rs 그대로
  │  broadcast::channel  │
  └──────────┬───────────┘
             │ subscribe
             ▼
  ┌──────────────────────┐
  │  monitor (MONITOR)   │  ← 새 바이너리, 일반 TCP 클라이언트로 접속
  │   ratatui TUI 렌더    │
  └──────────────────────┘
```

서버는 `if msg.addr == addr { continue; }`로 자기 자신에게는 메시지를 안 돌려보내기 때문에, 모니터는 자기 메시지를 빼고 **다른 모든 클라이언트의 트래픽을 깨끗하게** 받는다. 화면은 헤더 / 채팅 로그 / 입력창 / 키 안내 푸터의 4영역으로 구성되며, 채팅 로그는 최대 5,000줄까지 보관하면서 입퇴장은 노란색, 일반 메시지는 흰색, 내가 보낸 메시지는 초록색으로 색을 분리해 한눈에 흐름이 보이게 했다.

stress_test와 함께 띄우면 입장 알림 500개 → 일반 메시지 2,500개 → 퇴장 알림 500개가 약 4–5초 안에 폭포처럼 흘러내리는 게 그대로 보인다. 부하 테스트의 결과를 직접 보는 가장 직관적인 방법이다.

## 6. 결과 분석

### 6.1 핵심 지표 정리

| 지표 | 값 | 판정 | 비고 |
|------|------|------|------|
| 연결 성공 | 500 / 500 | ✅ | 재시도 로직 포함, 단 한 건도 실패 없음 |
| 연결 실패 | 0건 | ✅ | 배치 + 백오프 효과 |
| 총 송신 | 2,500건 | ✅ | 500 × 5, 예상치 일치 |
| 총 수신 | 88,667건 | ✅ | 3초 타임아웃 기준 |
| 처리량 | 535.6 msg/s | ✅ | 4.67초 동안 안정 유지 |
| 소요 시간 | 4.67초 | ✅ | 배치 지연 + 메시지 전파 포함 |

### 6.2 수신율 7.1 % 가 낮아 보이는 이유

언뜻 수신율이 낮아 보이지만, 이건 **서버 한계가 아니라 테스터의 설정 때문**이다. stress_test는 메시지 송신이 끝난 직후 3초만 더 기다리고 연결을 끊는다.

```
client_0000 의 타임라인

  T + 0    ms   연결 + 닉네임 전송
  T + 50   ms   1번째 메시지 송신
  T + 90   ms   5번째 메시지 송신
  T + 3090 ms   ⏱ 타임아웃 → 연결 종료
                ↑
                이 시점 이후에 도착하는 메시지 — broadcast 채널에서
                fan-out 중이거나 TCP 송신 버퍼에 대기 중이던 분량 —
                는 client_0000 의 RECV 카운터에 잡히지 않는다.
                → 카운팅 자체가 불가능

결론: 클라이언트가 연결을 유지한다고 가정하면 수신율은 사실상 100 %.
```

즉 서버는 메시지를 빠짐없이 전파하고 있고, 단지 *측정자*가 일찍 떠나는 것뿐이다.

### 6.3 무엇이 결과를 만들었나 — 4가지 설계 결정

500명에서 무실패가 가능했던 이유를 한 줄씩 정리하면 다음과 같다.

```
┌─────────────────────────────────────────────────────────────────────┐
│  1. 경량 Async Task                                                 │
│     OS 스레드(2MB 스택) 대신 tokio async task(수 KB) 사용           │
│     → 500개 연결 = 수 MB 메모리  (스레드 방식 대비 ~99% 절감)       │
├─────────────────────────────────────────────────────────────────────┤
│  2. Zero-Copy 브로드캐스트                                          │
│     broadcast::channel 은 Arc 기반 링 버퍼 사용                     │
│     → 클라이언트가 500명이어도 메시지 본체는 1회만 복사             │
├─────────────────────────────────────────────────────────────────────┤
│  3. Lock-Free 통계                                                  │
│     AtomicUsize::fetch_add(Relaxed) 로 카운터 집계                  │
│     → Mutex 없이 원자적 연산 → 카운터가 병목이 되지 않음            │
├─────────────────────────────────────────────────────────────────────┤
│  4. 교착 없는 I/O 분리                                              │
│     into_split() 으로 reader / writer 를 별도 소유권으로 분리       │
│     → 단방향 데이터 흐름 보장 → 구조적으로 deadlock 불가            │
└─────────────────────────────────────────────────────────────────────┘
```

이 4가지가 따로 놀지 않고 함께 작동했다는 점이 중요하다. 어느 하나라도 빠지면 어딘가에서 병목이나 버그로 이어졌을 가능성이 크다.

## 7. 마무리

### 7.1 한 줄 결론

> **500명이 동시에 들어와도 단톡방은 안 무너진다.**
> 연결 실패 0건, 처리량 535 msg/s, 4.67초 안에 깔끔히 처리.

### 7.2 아쉬운 점과 다음 과제

이번 구현은 단일 프로세스 안에서의 동시성 검증에 집중했기 때문에, 그대로 운영 환경에 올리려면 손볼 곳이 몇 가지 남아 있다.

- **수신율 100% 측정** — stress_test의 3초 타임아웃을 풀고, 메시지 ID 기반으로 정합성을 검증하면 이론상 1,247,500건 수신을 그대로 측정할 수 있다.
- **폭주 트래픽 대응** — broadcast 채널이 가득 찰 때 발생하는 `Lagged` 손실을 줄이기 위해, 클라이언트별 송신 큐 분리나 동적 용량 조정이 필요하다.
- **운영 환경 준비** — 평문 TCP는 학습용이지 운영용은 아니다. TLS 종단, 인증, 메시지 영속화(메시지 큐 / DB 적재) 계층이 추가되어야 한다.

### 7.3 배운 것

가장 크게 남은 교훈은 *동시성 문제는 코드보다 설계로 푸는 게 낫다*는 것이다. `into_split()`처럼 타입 차원에서 deadlock을 차단하거나, broadcast 채널 + 단일 송신자 구조처럼 데이터 흐름을 단방향으로 강제하는 결정들이, 결국 디버거로는 잡기 어려운 동시성 문제를 애초에 발생하지 않게 만들었다. 이 경험은 채팅 서버에 한정되지 않고 게임 서버·실시간 알림·IoT 메시지 허브처럼 fan-out 구조를 갖는 시스템 어디에든 그대로 적용할 수 있을 것 같다.
